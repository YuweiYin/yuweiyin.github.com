# Algorithm - Sort - String

Create Date: 2020.05.12

Last Update Date: 2020.05.12

By YuweiYin

# 简介

String 字符串相关数据结构与算法

String Matching 字符串匹配任务（子串匹配）。搜索引擎、文本编辑器的 Find 搜索功能、Unix 中的 grep 命令等各种应用都会使用到字符串匹配算法。

这里假设 文本 document 是一个长度为 n 的数组，T[0..n-1]，而 模式 pattern string 是一个长度为 m 的数组 P[0..m-1]，其中 m <= n，进一步假设 P 和 T 的元素都是来自于一个 有限字母集 \Sum 的字符。例如 \Sum = {0, 1} 或者 \Sum = {a, b, ..., z}。字符数组 P 和 T 通常被称为 字符串。

（后文也用 doc 表示长度为 n 的长文本字符串，ptn 表示长度为 m 的待匹配的模式字符串）

# 朴素字符串匹配算法

朴素字符串匹配算法是通过一个循环找到所有有效偏移，该循环对 n-m+1 个可能的子串起始位置 s 值进行检测，查看是否满足条件 P[0..m-1] = T[s..s+m-1]，伪代码如下：

NAIVE-STRING-MATCHER(T, P)
1 n = T.length
2 m = P.length
3 for s = 0 to n-m
4     if P[0..m] == T[s..s+m-1]
5         return s

时间复杂度为 O(m(n-m+1))

# Rabin-Karp Algorithm

在朴素字符串匹配算法的基础上，利用哈希散列技术，可设计出实际应用中效果良好的 Rabin-Karp 算法。该算法利用了初等数论的一些概念，比如两个数相对于第三个数模等价。

Rabin-Karp 算法需要进行 \Theta(m) 时间的预处理。在最坏情况下，整体的运行时间是 \Theta((n-m+1)m)，但是基于一些假设，在平均情况下，它的运行时间还是比较好的。

该算法将朴素匹配中检查字符串相等的操作，改为哈希表的增/删/查操作。朴素匹配中可以看作是一个长度为 m 的窗口滑过长度为 n 的文档，如果把这个滑动窗口看作是一个动态变化的哈希表，则可以将整体匹配的时间复杂度降到 O(n+m)。

尽管字符串的空间特别大，但是构成字符串的字符数量是比较有限的。

而且此算法处理全局查找（而不是仅找到第一个匹配的）时效率也很高。

观察：当哈希表在文档上滑动时，连续两步之间共享了 m-1 个相同字符，如果能够利用这个特性，让变化后的哈希表计算哈希值的时间（从本应的 O(m)）缩减为 O(1)，则能够达到迅速搜索的目标。

设计此哈希表为抽象数据类型 (Abstract Data Type, ADT)，名为 Rolling Hash Table，简写作 rht。

rht 维护一个字符串 x。rht 执行下面三个函数，时间复杂度均为 O(1)

  • rht.append(c) 往 x 尾部追加字符 c。模拟往右滑动时，窗口右侧新增的字符。
  • rht.skip() 删除 x 的首部字符。模拟往右滑动时，窗口左侧丢弃的字符。
    • 类似于 Python list 中的 pop(0) 或者 collections 中的 popleft()。
  • rht.h(x) 计算字符串 x 的哈希值。

算法思路如下：

  1. 实例化两个 rht 对象 r_doc 和 r_ptn
  2. 预处理（耗时 O(m)）：将 ptn 和首个窗口中的各字符转为数字
  3. 将模式串 ptn 中的每个字符加入到 r_ptn 中，计算出这 m 个字符组成的字符串（即 ptn）的哈希值 h_ptn = r_ptn.h(x)
  4. 将 doc 中的前 m 个字符添加到 r_doc 中，并计算出这前 m 个字符组成的字符串的哈希值 h_doc = r_doc.h(x)
  5. 循环：判断哈希值 h_ptn 是否等于 h_doc
    • A: if 如果相等
      • 此时只能表示两串的哈希值相同，不一定代表这两串一定相同，因为可能会有碰撞冲突。
      • 因此此时需要逐字符检测是否匹配，耗时 O(m)。如果此时匹配成功，函数返回 return；否则往下执行 B 过程
    • B: if 如果不相等，且检索到了尾部，匹配失败，跳出循环 break
    • C: else 否则
      • 从 r_doc 中移除最左字符，r_doc.skip()
      • 在右侧追加一个字符，r_doc.append(next_c)
      • 重新计算哈希值，h_doc = r_doc.h(x')，继续循环 loop
      • （核心是要利用重复的信息，保证在 O(1) 时间内计算新串的哈希值）
  6. 匹配成功则返回模式串所在起始下标，否则返回 -1

因此最坏情况是：长文本 doc 中 所有窗口里的字符串哈希值都和 ptn 模式串的哈希值相等，但是它们实际又不完全逐字符完全匹配，这样每个窗口都要耗费 O(m) 的时间去逐字符匹配，导致总时间为 O((n-m+1)m)。但只要处理好哈希函数，就能很大程度减少冲突（让冲突出现的概率不超过 1/m，这样逐字符匹配的期望时间就是 O(1)），从而达到 Rabin-Karp 算法的平均时间复杂度 O(n+m)。

Rabin 和 Karp 证明了，即便是最简单的 Division Hashing 除法散列哈希函数 h(k) = k \mod p，只要 p 是大于等于 m（模式串长度）的 random prime 随机选择的素数，那么此算法就能高效执行。而且 p 选择得越大，哈希冲突出现的概率就越小。

一般而言，模式串 ptn 不会太长，所以大于 m 的素数是容易找到的。至于寻找相当大的素数，可以参考《CLRS》第 31 章数论算法中的 31.8 节素数的测试。
